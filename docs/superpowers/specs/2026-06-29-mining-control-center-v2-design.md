# مركز التحكم بالتعدين — v2 (الحساب والمزامنة)
### وثيقة التصميم
- **التاريخ:** 2026-06-29
- **الحالة:** مسودة بانتظار موافقة المستخدم
- **يبني على:** v1 (`2026-06-29-mining-control-center-design.md`) ويعيد استخدام منطقها بالكامل.

---

## 1. الهدف
برنامج يُثبَّت على عدة أجهزة، فيه **حساب وتسجيل دخول**. عند الدخول بنفس الحساب من أي جهاز
تظهر **كل أجهزة الأسيك من كل المواقع** مجمّعة، مع **مراقبة** و**تحكّم** (إعادة تشغيل/إيقاف/تشغيل/Reboot
و**تغيير البول**). الوصول للأسيكات يتم عبر **وكيل محلي** في كل موقع — بلا فتح منافذ ولا كشف للأجهزة.

## 2. لماذا تغيّرنا عن v1
- أسيكات كل موقع خلف راوتر **لا يصل إليه المستخدم مباشرة**، لكن في كل موقع **لابتوب شغّال** على نفس شبكة الأسيكات.
- المطلوب «دخول بنفس الحساب من أي جهاز ورؤية الكل» يستلزم **خادماً مركزياً** كنقطة تجمّع.
- هذا **يلغي** الحاجة لفتح المنافذ/DDNS من v1: الوكيل المحلي يكلّم الأسيكات على الـ LAN، ويتصل **صادراً** بالخادم.

## 3. المعمارية
```
 موقع: لابتوب [وضع وكيل] ──(wss صادر)──┐
   └─ Antminer×N (192.168.x، محلي)      │      ┌─ جهازك [وضع عارض]
                                  [ خادم VPS ]──┤
 موقع آخر: لابتوب [وكيل] ──(wss صادر)──┘       └─ جوال/لابتوب آخر [عارض]
```
- **نفس التطبيق، وضعان:**
  - **وكيل (Agent):** على لابتوب الموقع. يضيف أسيكات الموقع بعناوينها المحلية، يراقبها، يدفع حالتها للخادم، وينفّذ الأوامر القادمة محلياً.
  - **عارض (Viewer):** على جهاز المستخدم. يعرض كل أجهزة الحساب ويرسل الأوامر.
  - أي جهاز يقدر يكون الاثنين (يضيف أجهزة محلية ويشوف الكل).
- **الخادم (VPS):** حسابات + تخزين + **توجيه الأوامر** (عارض → خادم → الوكيل المالك للجهاز → الأسيك → نتيجة)، كله عبر WebSocket مشفّر.

## 4. التقنية
- **الخادم:** Node.js + TypeScript + WebSocket (`ws`) + **SQLite على السيرفر** (Node عادي، بلا أي تعقيد native) + REST خفيف للمصادقة.
- **المصادقة:** بريد + كلمة مرور، تجزئة `bcrypt`، جلسة **JWT** (للـ REST و WS).
- **التشفير/النقل:** **TLS بشهادة ذاتية** على IP الـ VPS (لا دومين). التطبيق **يثبّت بصمة الشهادة (pinning)** للخادم الذي يدخله المستخدم.
- **التطبيق:** Electron + React الحالي + طبقة عميل للخادم (REST + WS).
- **متعدد المستخدمين:** كل حساب يرى بياناته فقط (عزل بـ `userId`). صلاحيات متقدمة مؤجّلة.

## 5. نموذج البيانات (SQLite على الخادم)
- `users`: id, email, passwordHash, createdAt
- `agents`: id, userId, name, lastSeenAt (كل تثبيت بوضع وكيل = سجل واحد، له token)
- `sites`: id, userId, name
- `devices`: id, userId, siteId, agentId, name, model, firmware, host(محلي), apiPort, controlPort
  - **سرّ الدخول للأسيك يبقى محلياً على الوكيل فقط** (لا يُرفع للخادم) — أأمن.
- `device_status` (آخر حالة، تُحدّث دورياً): deviceId, state, hashrate, temp, fan, pool, worker, uptime, updatedAt

## 6. تدفّق المصادقة
1. **تسجيل:** `POST /auth/signup {email,password}` → ينشئ مستخدماً.
2. **دخول:** `POST /auth/login {email,password}` → يعيد JWT.
3. كل اتصال WS وREST يحمل الـ JWT؛ الخادم يتحقق ويربط الطلب بـ `userId`.

## 7. بروتوكول المزامنة (WebSocket فوق TLS)
- **الوكيل → الخادم:** `agent.hello {agentId,name}` · `device.register {device(بدون سر)}` · `status.update {deviceId,status}` (مجمّعة) · `command.result {commandId,outcome}`.
- **الخادم → الوكيل:** `command.exec {commandId,deviceId,command,params}` (يطابق أمراً من عارض).
- **العارض → الخادم:** `snapshot.request` · `command.send {deviceId,command,params}`.
- **الخادم → العارض:** `snapshot {sites,devices,statuses}` · `status.update {...}` (بثّ) · `command.ack {commandId,outcome}`.
- الخادم يحتفظ بخريطة `agentId → ws`؛ عند أمر من عارض يجد `device.agentId` ويوجّه `command.exec` للوكيل، ينتظر `command.result`، ويعيدها للعارض.
- إن كان الوكيل غير متصل → الأجهزة تظهر «الوكيل غير متصل».

## 8. إعادة استخدام منطق v1
- داخل **الوكيل**: نفس `cgminer/4028` + المحوّلات + `monitor` + `bulk` + النقل TCP/HTTP. السر يُقرأ محلياً (DPAPI كما هو).
- داخل **العارض**: نفس واجهة React (الجدول المضغوط المقسّم حسب الموقع)، لكن مصدر البيانات يصير الخادم بدل المحلي.

## 9. ميزة جديدة: تغيير البول (Change Pool)
- أمر تحكّم جديد `setPool {url, user, pass}` يُضاف لواجهة `DeviceDriver` وكل محوّل:
  - **Stock:** عبر `cgi-bin/set_miner_conf.cgi` (POST تهيئة) — قد يحتاج ضبط لكل موديل.
  - **Braiins / Vnish / LuxOS:** عبر واجهة الفرمور الخاصة (إضافة/تبديل pool).
- يُتحقق عملياً على أجهزة حقيقية (يختلف بين الموديلات — يُسجّل كخطر).

## 10. الأمان
- TLS بشهادة ذاتية + **تثبيت البصمة** في التطبيق (يمنع اعتراض الاتصال).
- كلمات مرور الحسابات بـ bcrypt؛ JWT بصلاحية وتجديد.
- **أسرار الأسيكات لا تغادر الوكيل** (تبقى محلية مشفّرة).
- عزل بيانات كل حساب على الخادم؛ كل وكيل/عارض يصل لبيانات حسابه فقط.
- الأسيكات لا تُكشف للإنترنت إطلاقاً (الوكيل المحلي فقط يكلّمها).

## 11. النطاق
**داخل v2:**
- خادم: مصادقة (signup/login) + تخزين + توجيه أوامر/حالة عبر WS + TLS ذاتي.
- التطبيق: شاشة دخول/تسجيل، اختيار الوضع (وكيل/عارض)، وضع الوكيل (تسجيل أجهزة + تنفيذ أوامر + دفع حالة)، وضع العارض (عرض الكل + إرسال أوامر).
- أمر **تغيير البول** للمحوّلات الأربعة.
- دليل نشر الخادم على VPS.

**مؤجّل:**
- صلاحيات/أدوار دقيقة بين المستخدمين، رسوم بيانية تاريخية، تطبيق جوال أصلي (العارض يكفي عبر الويب/الديسكتوب)، تحديث فرمور، auto-tuning.

## 12. المخاطر والافتراضات
- **شهادة ذاتية بلا دومين:** يتطلب تثبيت بصمة في التطبيق؛ تغيّر IP الـ VPS يتطلب تحديث. (دومين لاحقاً يبسّط الأمر.)
- **تغيير البول** يختلف بين موديلات/فرمور — يحتاج تحقق ميداني.
- **الوكيل = نقطة الموقع:** إطفاء لابتوب الموقع = أجهزته غير قابلة للتحكم حتى يعود (متوقّع).
- المستخدم سيوفّر VPS (وصول SSH) لنشر الخادم؛ النشر موثّق.

## 13. الخطوة التالية
بعد موافقة المستخدم → **خطة تنفيذ** مرحلية (writing-plans): الخادم أولاً (مصادقة + WS + تخزين)، ثم وضع الوكيل، ثم وضع العارض، ثم تغيير البول، ثم النشر.
